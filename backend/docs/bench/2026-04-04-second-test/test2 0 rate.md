## 深度分析与结论
1. **吞吐量翻倍 (306 QPS)**：
   - 相比第二次压测的 147 QPS，性能提升了 **108%**。这证明了在异步化架构下，通过调整压测参数（Rate=0）能压榨出单机更高的处理潜力。
2. **库存售罄点 (9332 Success)**：
   - 成功下单 9332 次，说明系统在崩溃前几乎已经完成了所有有效库存的分配。
3. **连接拒绝错误 (Code 0: 9978次)**：
   - 错误：`connectex: No connection could be made...`
   - **分析**：这仍然是单机环境的物理瓶颈。当 QPS 冲上 600+ 时，操作系统的临时端口耗尽或 API 服务的连接队列溢出，导致约一半的请求被拒。
4. **长尾延迟 (p99 11s)**：
   - 随着 QPS 升高，请求在 Redis 队列和网络缓冲区堆积，导致尾部延迟显著增加。

## 改进分析
1.redis中stock还剩593，出现了少卖的现象，redis的poolsize为500，分析原因是系统端口耗尽，AI分析可用一下方法：
- 系统参数调优 ：在注册表中调大 MaxUserPort 并调小 TcpTimedWaitDelay 以加速端口回收。
- 长连接复用 ：确保压测工具和后端都开启了 Keep-Alive ，减少频繁的 TCP 握手。
- 负载均衡 ：最根本的解决方式是 ‘横向扩展’ ，通过增加 API 节点数量，将流量分散到不同的 IP/端口上，从而避开单机内核的队列上限。”
